# EXPERIENCE.md — Stage 1 经验积累

> **这不是教程，这是经历。**
> 
> 每一条都是真实的失败或成功案例，总结出来的判断依据。
> 激活 Stage 1 时必读，作为隐式决策的基础。

---

## EXP-001: 快速识别"模糊需求"的两个信号

### 见过
用户说"帮我分析一下""帮我写代码""帮我生图"，然后我直接问"要用什么语言"，用户卡住了——说"不知道，看你的"。

### 因此知道
如果用户的回答包含以下模式，说明需求还在脑子里没整理清楚：
- "随便" / "都可以" / "看你的"
- "普通的" / "一般的" / "标准的"
- 重复说同一个词但没有细节（"分析分析"）
- 问我"应该怎么做"而不是说"我想怎么做"

### 现在会
不要硬推进。这时候要**从侧面追问**，不是直接问"什么类型"。改成：
```
"理解。但我想确认一个细节——你想看什么样的结果？
 比如说，是想看数据趋势、对比对比、还是挖掘隐藏的规律？"
```

这样问法好处：给了用户**可选框架**，让他能pick，而不是强行让他创造。

## EXP-002: 不同任务类型的"真实差异"——不要用一套问法

### 见过
我用"你的需求是什么""有什么约束吗"这套万能问法，结果：
- 对数据分析很好用（他们能列出数据源、指标、约束）
- 对文案创作效果差（他们开始纠结"是否应该用比喻"而不是说受众）

### 因此知道
**不同任务类型的信息优先级完全不同**：

| 任务类型 | 第一优先级 | 第二优先级 | 第三优先级 |
|---------|----------|----------|----------|
| **数据分析** | 数据源 | 操作类型 | 输出格式 |
| **代码生成** | 语言/框架 | 具体功能 | 错误处理 |
| **文案** | 受众 | 品牌调性 | 转化目标 |
| **绘画** | 风格 | 主体 | 色调 |

问错顺序的后果：用户容易跑题。比如问数据分析师"你想要什么风格的报告"，他反而困惑。

### 现在会
**快速识别类型后，立即切换问法框架**。不浪费时间问无关的问题。

## EXP-003: "理解确认"这一步真的必须有

### 见过
有一次我跳过了确认，直接生成。用户回来说"这不是我想要的"。原因是我理解成了"周期性数据"，实际他要的是"一次性快照"。

浪费了 10 分钟重做。

### 因此知道
**理解确认不是浪费时间，是防呆机制**。这一步做得好，后续的 Stage 2-5 都会顺利。反之会产生一连串问题。

### 现在会
在进入生成前，**必须用"我的理解是..."这个模板确认**。格式：
```
我的理解：
- 核心功能：[一句话总结]
- 关键约束：[2-3 个]
- 成功标准：[如何判断对了]

对吗？
```

用户要是说"对，就是这样"才能进入生成。说"差不多但..."就再补充细节再确认。

## EXP-004: 版本决策树准确度的秘密——参数权重很重要

### 见过
我最初的决策树是简单的投票制：
```
if 明确度 + 约束数 + 复杂度 >= 某个阈值 → 详细版
```

结果：有些复杂但约束少的任务，我生成了过度详细的提示词（反而让用户困惑）。

### 因此知道
**复杂度的权重应该最高**。原因：
- 复杂任务即使约束少，也需要详细的分步
- 简单任务即使约束多，也可以简洁表述

### 现在会
调整权重：
```
score = 明确度 * 0.3 + (1 - 约束数/10) * 0.2 + 复杂度 * 0.5
```

这样复杂度占了主导，更符合实际。

## EXP-005: 模糊回答的"真正原因"不是用户不专业，是问法不对

### 见过
我问"你想要什么代码"，用户说"普通的就行"。
我当时觉得他不专业。后来重新问"Web 开发还是数据处理？"他立即说"哦，Web，用 React"。

### 因此知道
**用户不是不能说清，是问题给的框架太宽**。"什么代码"的范围从游戏到操作系统，他当然说不清。

### 现在会
提问时**总是提供 2-3 个具体选项**：
```
✗ 错：你想要什么代码？
✓ 对：是 Web 应用（React/Vue）、数据处理（Python）、还是其他？
```

这样用户能 pick，不需要凭空想象。

## EXP-006: 跨域任务的真实处理方式——不要硬分类

### 见过
用户说"帮我写一个 SKILL，既能分析数据，又能生成报告文案"。
我硬是把它分成"数据任务"和"文案任务"，结果用户说"不对，这是一个整体"。

### 因此知道
**不是所有任务都能清晰分类**。有时候跨域就是真实需求。

### 现在会
识别到跨域后，**确认主路径**：
```
"我看到这个任务涉及数据处理和文案。
如果只能选一个作为主要方向，你会选哪个？"
```

用户选定主路径后，其他部分作为"集成需求"补充说明。

## EXP-007: 成功率预估的标准——不要瞎编，要基于维度

### 见过
我最初的做法是拍脑袋："这个看起来不错，成功率 80%"。结果用户到 Stage 2 才发现问题。

### 因此知道
**成功率要基于三个维度的实际值**：

```
成功率 = 明确度 * 100% （如果 >= 0.8）
       = (明确度 * 0.6 + 约束完整度 * 0.4) * 100% （如果 0.6-0.8）
       = (明确度 * 0.5 + 约束 * 0.3 + 我的理解确认次数 * 0.2) * 100% （如果 < 0.6）
```

### 现在会
每生成一个提示词后，**实际计算一下成功率**，而不是估计。这样 Stage 2 的审核会更有针对性。

## EXP-008: "生成版本"和"成功率"的关联

### 见过
有一次我生成了"简洁版"（因为需求很清楚），但成功率标成了 60%。用户看到这个数字就害怕了："你都不确定为什么给简洁版？"

### 因此知道
**版本和成功率必须对应**：
- 简洁版 → 成功率应该 >= 85%
- 标准版 → 成功率应该 70-85%
- 详细版 → 成功率可以 50-70%（因为细节我自己补了）

如果版本和成功率矛盾，说明我判断有问题。

### 现在会
生成前**做一遍检查**：版本和成功率是否对得上。如果不对，重新审视需求。

## EXP-009: 提示词模板的"通用性陷阱"

### 见过
我给所有任务都用一个模板：
```
# [标题]
## 核心职责
## 步骤
## 输入输出
## 边界
```

结果对"简单数据聚合"太啰嗦，对"复杂 API 设计"又不够详细。

### 因此知道
**不同复杂度的任务需要不同的模板深度**：

- **低复杂度**：只需 title + 步骤 + 示例，两页纸之内
- **中复杂度**：+ 约束条件 + 错误处理
- **高复杂度**：+ 详细设计 + 伪代码 + 性能考虑

### 现在会
生成时**自动选择模板复杂度**，而不是所有任务一个样子。

## EXP-010: 交付给 Stage 2 时的"完整性检查清单"

### 见过
有一次我生成了提示词，但没有说这是什么任务类型、什么复杂度。Stage 2 审核时要猜测，浪费了时间。

### 因此知道
**交付时必须清楚说明**，让 Stage 2 能快速上下文切换。

### 现在会
交付格式固定化：
```
📝 SKILL 提示词生成完成

【任务信息】
- 类型：[数据/代码/文案/绘画/其他]
- 复杂度：[低/中/高]
- 版本：[简洁/标准/详细]

【质量指标】
- 需求明确度：XX%
- 约束完整度：XX%
- 预估成功率：XX%

【关键决策点】
- [如果有特殊的理解或假设，列出来]

【下一步】
已准备好提交 Stage 2 安全审核。
```

这样 Stage 2 一看就知道怎么审。
