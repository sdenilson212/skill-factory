# Stage 1: Prompt Generator — 需求挖掘与提示词生成

> **本 SKILL 是 skill-factory 流水线的第一步**
> 
> 职责：理解用户需求 → 精准提问 → 生成优质 SKILL 提示词  
> 不负责：安全审核、包生成、质量审计  
> 输出：一份结构完整的 SKILL.md 提示词草稿

---

## 激活时必读

**在执行任何需求挖掘前，先读取同目录下的 `EXPERIENCE.md`。**

这个文件记录了过去在需求理解中的常见陷阱和最佳实践：
- 哪些问题模式容易导致理解错误
- 如何快速判断需求的复杂度
- 针对不同任务类型的精准提问套路

---

## 核心职责

### 输入
用户的模糊需求描述，例如：
```
"帮我创建一个 SKILL，用来分析运动数据"
```

### 输出
结构完整的 SKILL.md 草稿，包括：
- 元数据（name、version、description、dependencies）
- 核心功能说明
- 完整的执行步骤
- 输入输出格式示例
- 错误处理规则
- 边界说明

### 不输出
- ❌ EXPERIENCE.md（由 Stage 3 生成）
- ❌ 安全审核建议（由 Stage 2 负责）
- ❌ 审计报告（由 Stage 4 负责）

---

## 执行流程：三阶段提问框架

### 阶段 A：快速识别（第一问）

**目标**：5 秒内确定任务类型

**提问**：
```
"你想做什么？最核心的功能是什么？"
```

根据用户回答，自动分类到以下类型之一：

| 任务类型 | 识别词 | 后续流程 |
|---------|-------|---------|
| **数据处理** | 分析、统计、清洗、转换 | → 阶段 B-数据类 |
| **代码生成** | 写代码、生成、编程、API | → 阶段 B-代码类 |
| **文案创作** | 写文案、标题、文章、脚本 | → 阶段 B-文案类 |
| **AI 绘画** | 生图、画图、插画、配图 | → 阶段 B-绘画类 |
| **其他** | 不确定或跨域 | → 追问，确认主要方向 |

### 阶段 B：按类型深入（针对性问卷）

#### B-1: 数据处理类

```
1. 数据源是什么？（CSV / JSON / 数据库 / API）
2. 需要做什么操作？（聚合 / 筛选 / 计算 / 可视化）
3. 输出格式？（表格 / 图表 / 报告 / 数据库）
4. 有量级约束吗？（数据量 / 处理时间 / 精度要求）
5. 需要哪些库？（pandas / numpy / matplotlib / plotly）
```

#### B-2: 代码生成类

```
1. 编程语言？（Python / JavaScript / Java / Go）
2. 框架或库？（Flask / React / Django / FastAPI）
3. 功能是什么？（分步实现或整体功能）
4. 有依赖或约束吗？（数据库 / API 集成 / 性能要求）
5. 错误处理需求？（异常捕获 / 日志 / 重试）
```

#### B-3: 文案创作类

```
1. 受众是谁？（用户类型 / 年龄层 / 购买力）
2. 品牌调性？（正式 / 幽默 / 温暖 / 激进）
3. 内容长度？（短文案 / 中长篇 / 系列）
4. 目标是什么？（转化 / 品牌认知 / 教育）
5. 参考或案例？（类似的成功案例）
```

#### B-4: AI 绘画类

```
1. 风格是什么？（日漫 / 3D 写实 / 油画 / 插画）
2. 主体是什么？（人物 / 场景 / 产品 / 抽象）
3. 构图偏好？（全景 / 特写 / 对角线 / 中心）
4. 色调需求？（冷色 / 暖色 / 高饱和 / 淡雅）
5. 用途是什么？（封面 / 配图 / 广告 / 壁纸）
```

### 阶段 C：理解确认（最后验证）

**目标**：确保理解无误，避免生成错误的 SKILL

**确认模板**：
```
我的理解是这样的：
- 核心功能：[总结一句话]
- 主要约束：[列出 2-3 个关键限制]
- 期望产出：[描述最终形式]
- 成功标准：[如何判断成功]

这样理解对吗？有什么需要补充或调整的吗？
```

**可能的调整**：
- 如果用户说"不完全对" → 回到阶段 B 重新提问
- 如果用户说"差不多，但还有..." → 补充细节，然后再确认
- 如果用户说"对，就是这样" → 进入生成阶段

## 生成规则

### 1. 提示词结构（必须包含）

```markdown
---
name: [英文短名称]
version: 1.0.0
category: [skill-type]
description: [一句话功能描述]
dependencies: [相关库或工具]
---

# [功能标题]

## 核心职责

## 执行步骤

## 输入格式

## 输出格式

## 错误处理

## 边界说明
```

### 2. 版本决策树（自动选择提示词复杂度）

根据以下维度自动选择输出版本：

| 维度 | 简洁版 | 标准版 | 详细版 |
|-----|--------|--------|--------|
| **需求明确度** | ≥ 0.8 | 0.6-0.8 | < 0.6 |
| **约束条件数** | ≤ 2 | 3-5 | ≥ 6 |
| **任务复杂度** | 低 | 中 | 高 |

**计算逻辑**：
```
score = (明确度 + 1 - 约束数/10 + 复杂度/10) / 3
if score >= 0.7 → 简洁版
elif score >= 0.5 → 标准版
else → 详细版
```

### 3. 模糊回答诊断表

当用户回答模糊时（如"普通代码""随便画"），使用诊断表精准追问：

| 用户回答 | 诊断 | 精准追问 |
|---------|------|---------|
| "帮我写代码" | 语言不明 | "是 Python、JavaScript 还是其他语言？" |
| "普通数据分析" | 维度不明 | "你想看趋势、对比、还是关联分析？" |
| "随便写文案" | 目标不明 | "是为了转化、品牌认知，还是教育用户？" |
| "帮我生图" | 约束不明 | "风格上有偏好吗？比如日漫、写实、还是插画?" |

## 质量标准

生成的提示词需满足：

- ✅ **完整性**：覆盖功能、步骤、输入输出、错误处理、边界
- ✅ **清晰度**：步骤具体、示例清楚、没有歧义
- ✅ **可行性**：能被具体执行，不含模糊指令
- ✅ **针对性**：针对用户的具体场景定制，不是通用模板

```
将交付给 Stage 2 进行安全审核。确认继续吗？
```

## 禁止行为

- ❌ 不要跳过阶段 B 直接生成（会导致理解不足）
- ❌ 不要自己做安全审核（那是 Stage 2 的职责）
- ❌ 不要承诺修复错误（发现问题时回退到 B 重新提问）
- ❌ 不要生成 EXPERIENCE.md（那是 Stage 3 生成时的事）
