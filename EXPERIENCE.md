# EXPERIENCE.md — skill-factory 的经历积累

> 三段格式：见过 → 因此知道 → 现在会
> 每条经验必须有具体情境，禁止泛化描述。

---

## 第一部分：结构性经验（制作时写入）

## EXP-001：依赖声明与实际用途不匹配

**见过**：skill-factory v1.0.1 在 frontmatter 的 `dependencies` 里声明了 `prompt-auditor`，Stage 0 也检查它是否安装，但实际上 Stage 2 做的是 SKILL.md 结构完整性审核，而 `prompt-auditor` 的职责是提示词安全注入检测——两者根本不是同一件事。用户在 2026-03-31 指出这个问题：「prompt-auditor 是提示词审计不是 SKILL.md 安全性审核」。

**因此知道**：声明了依赖但没有真正激活，或者激活的工具和当前任务不对口，这是流水线最容易出现的"名不副实"问题。依赖越多，这类误差越难发现，因为没人会每次都核对依赖列表和实际步骤是否一致。

**现在会**：在生成新 SKILL 的 frontmatter dependencies 之前，先逐条对照：这个依赖在哪个 Stage 被真正激活？激活后用来做什么？如果找不到对应步骤，就删掉它——不需要的依赖比没有依赖更有害，因为它会误导安装检查。

---

## EXP-002：Stage 0 检查与 frontmatter 依赖必须严格同步

**见过**：v1.0.1 修复了 frontmatter 把 `prompt-auditor` 去掉，但 Stage 0 的检查目录列表（四个目录）和阶段输出格式里仍然保留了 `prompt-auditor ✓` 这一行——两处不同步，导致检查通过了一个不再需要的依赖。

**因此知道**：流水线类的 SKILL 有多个地方描述同一份依赖清单（frontmatter / Stage 0 检查列表 / Stage 0 输出格式），只改一处必然漏改。任何涉及"列表"的修改都要做全局搜索，确认同一内容出现了几处。

**现在会**：修改依赖时，用全文搜索定位所有出现该依赖名称的位置，逐一确认是否需要同步修改，改完之后再做一次全文搜索验证没有遗漏。

---

## EXP-003：提示词草稿的审核工具选择不能靠"感觉相关"

**见过**：`prompt-auditor` 名字里有"审核"两个字，`skill-factory` 的 Stage 2 也叫"审核提示词"，这两者在名字上高度相似，导致最初写流水线时把 `prompt-auditor` 挂在了 Stage 2。但 `prompt-auditor` 实际检查的是提示词中的安全注入风险（越狱、系统信息泄露），而 Stage 2 需要的是 SKILL.md 结构完整性审核（步骤数、示例、frontmatter 格式）。

**因此知道**：工具的命名往往比实际职责更宽泛。"名字相似 ≠ 职责相同"，选工具之前必须先看它的 SKILL.md 里的"解决什么问题"段，而不是从名字猜测。

**现在会**：在 Stage 2 或任何需要引用外部工具的步骤前，先翻开该工具的 SKILL.md，找到"触发场景"或"功能描述"段，确认它的输入输出是否和当前任务匹配，再决定用不用。

---

## EXP-004：EXPERIENCE.md 的写入时机决定它的价值

**见过**：skill-factory 在 Stage 3 规定"同步生成 EXPERIENCE.md，写入 3～5 条结构性经验"，但这一步很容易被执行者用通用占位符敷衍填满——比如"见过：曾有人在做 SKILL 时遇到问题"这类没有具体情境的泛化内容。这类内容被加载后对判断没有任何帮助，等同于空文件。

**因此知道**：EXPERIENCE.md 的三段结构（见过/因此知道/现在会）只有在填入**真实发生的、有时间和场景的事件**时才有意义。写入时机很关键：在流程正式运行、出现了真实情况后写，而不是在"什么都还没发生"时预先凑条数。

**现在会**：Stage 3 生成 EXPERIENCE.md 时，只写能在当前上下文里举出真实例子的条目；如果当前 SKILL 是首次制作、还没有真实案例，第一部分最多写 2 条（聚焦在设计决策上），其余留给第二部分自然积累，禁止编造场景凑够 3～5 条。
