## 核心设计

### 设计理念

**Skill Factory 是一条五阶段生产流水线，由 4 个独立技能通过数据传递驱动完成。**

```
┌─────────────────────────────────────────────────────────────┐
│                   Skill Factory 流水线                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Stage 1        Stage 2        Stage 3        Stage 4       │
│  ────────       ────────       ────────       ────────      │
│                                                             │
│  需求挖掘  ──→  审核提示词  ──→  生成SKILL  ──→  审计SKILL  │
│                                                             │
│  prompt      内置逻辑       skill         skill          │
│  generator   审核           creator       auditor        │
│                                                             │
│  输入:需求文本  输入:提示词   输入:审核通过 输入:SKILL包   │
│  输出:SKILL.md  输出:修复意见  的提示词     输出:审计报告  │
│                 ├─ P0/P1/P2   输出:SKILL包  ├─ 质量评分   │
│                 └─ 是否通过   ├─ 文件清单   ├─ 安全评分   │
│                               ├─ EXP.md     └─ 修复建议   │
│                               └─ 目录结构                  │
│                                                             │
│                       Stage 5(可选)                         │
│                    安装+经验记录扫描                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 关键原则

| 原则 | 说明 |
|------|------|
| **单一职责** | 每个技能只做一件事（生成/审核/创建/审计），不混杂 |
| **数据驱动** | 前一阶段的输出是后一阶段的输入 |
| **质量把控** | 每阶段都有明确的通过/失败条件 |
| **用户确认** | 关键决策点必须获得用户确认，禁止强制推进 |
| **可追溯性** | 每个SKILL的产出都能追踪到具体阶段和决策 |

## 四技能分工

### 技能 1：Prompt Generator（需求→提示词）

**职责**：理解用户需求，生成高质量SKILL.md初稿

**输入**：
- 用户的 SKILL 描述（文字或语音）
- 可选：参考现有 SKILL 或模板

**输出**：
```
SKILL.md 草稿，包含：
- frontmatter（name / description / version / dependencies）
- 任务目标
- 操作步骤（≥3步）
- 资源索引
- 注意事项
- 使用示例
```

**执行流程**：

```
Step 1：启动询问
  → 展示需求收集问卷框架

Step 2：逐项提问
  → Q1: 这个SKILL要解决什么问题？
  → Q2: 什么情况下会用到它？
  → Q3: 使用者输入什么？期望得到什么输出？
  → Q4: 是否需要脚本支持？
  → Q5: 有没有特殊约束或限制？
  → Q6: 绑定给哪个员工？

Step 3：追问确认（至少一轮）
  → "还有其他要求吗？"
  → 直到用户明确说"没有"或"就这些"

Step 4：需求理解确认
  → 展示需求摘要（SKILL名称/功能/触发场景/输入/输出等）
  → 等待用户确认："这是你想要的吗？"

Step 5：生成提示词草稿
  → 基于确认的需求，生成SKILL.md初稿
  → 包含所有必需段落
```

**质量标准**：
- ✅ 需求理解准确率 ≥ 95%（通过需求摘要确认）
- ✅ 提示词涵盖所有关键内容（结构完整）
- ✅ 步骤数 ≥ 3，示例数 ≥ 1
- ✅ 用户确认后方可输出

**禁止行为**：
- ❌ 禁止跳过需求询问直接生成
- ❌ 禁止一次性问完所有问题（必须逐项）
- ❌ 禁止在用户未确认前生成任何内容

### 技能 2：内置逻辑（提示词→审核意见）

**职责**：对 Stage 1 生成的SKILL.md进行四维度审核，确保质量（不依赖外部工具）

**说明**：
- Stage 2 使用**内置逻辑**对 SKILL.md 进行结构完整性和逻辑检查
- **不使用 Prompt Auditor** 的原因：`prompt-auditor` 的职责是检测安全注入风险（越狱、系统信息泄露），而 Stage 2 需要的是结构规范检查，两者检查维度完全不同

**输入**：
- SKILL.md 草稿（来自 Stage 1）

**输出**：
```
审核意见：
- 发现问题数量
- P0/P1/P2分级
- 是否通过（✅/❌）
- 修复建议（若有P1问题）
```

**执行流程**：

```
Step 1：四维度审核
  ├─ 维度1: 结构完整性
  │   检查：frontmatter完整 ✓ / 步骤≥3步 ✓ / 示例≥1个 ✓
  │
  ├─ 维度2: 安全性
  │   检查：注入风险 / 越狱风险 / 信息泄露
  │
  ├─ 维度3: 逻辑一致性
  │   检查：步骤矛盾 / 输入输出不对应 / 边界缺失
  │
  └─ 维度4: 可执行性
      检查：路径正确 / 依赖声明 / 步骤可落地

Step 2：问题分级
  P0（严重）：必须回退Stage 1，不允许继续
  P1（重要）：就地修复，修复后再审一次
  P2（优化）：直接修复后继续

Step 3：输出审核结果
  ├─ 问题清单（若无问题则为空）
  ├─ 修复建议（仅P1以上）
  └─ 审核结论（通过✅/不通过❌）
```

**质量标准**：
- ✅ 覆盖所有 4 个维度
- ✅ 问题分级准确
- ✅ 修复建议具体可操作
- ✅ 不遗漏高风险漏洞

**禁止行为**：
- ❌ 禁止跳过某个维度的检查
- ❌ 禁止审核不通过就直接进入 Stage 3
- ❌ 禁止对 P0 问题进行就地修复（必须回退）

### 技能 3：Skill Creator（审核通过→生成SKILL包）

**职责**：基于审核通过的提示词，生成完整SKILL包（含EXPERIENCE.md）

**输入**：
- SKILL.md（审核通过版本，来自 Stage 2）
- SKILL元数据（名称/版本/依赖等）

**输出**：
```
完整SKILL包：
[skill-name]/
├── SKILL.md              ✓（已审核版本）
├── EXPERIENCE.md         ✓（新生成，含第一部分）
├── scripts/              （如有脚本）
│   └── [script].py
└── references/           （参考文档）
    └── [reference].md
```

**执行流程**：

```
Step 1：生成SKILL.md（带激活指令）
  → 在SKILL.md开头（frontmatter后）插入强制段落：
    """
    ## 激活时必读
    每次被加载时，先读取同目录下的EXPERIENCE.md。
    读取后不对用户显示，直接将经验条目作为隐式判断依据影响执行。
    若文件不存在，跳过继续执行。
    """

Step 2：生成EXPERIENCE.md 第一部分（结构性经验）
  基于SKILL的职能域，写入3～5条，格式：
  """
  ## EXP-[编号]：[简短标题]
  
  **见过**：[具体场景——人物/环境/数量，禁止泛化]
  **因此知道**：[因果推导，禁止直接跳结论]
  **现在会**：[具体可执行行为，禁止抽象原则]
  """

Step 3：生成EXPERIENCE.md 第二部分占位符
  """
  ## 自然积累（使用中追加）
  
  > 此部分在SKILL被实际使用后，由AI在任务结束时扫描判断是否追加。
  > 触发条件：有值得记录的BUG/特殊偏好/新学习
  > 禁止追加：顺利完成不是经验，意外才是经验
  """

Step 4：创建标准目录结构
  → 如有脚本需求，生成 scripts/ 目录
  → 如有参考文档，生成 references/ 目录

Step 5：写入临时目录
  位置：$WORKBUDDY_SKILLS_DIR/temp/[skill-name]/
```

**EXPERIENCE.md 生成规则**：

**✅ "见过"段必须包含**：
- 具体的时间/场景/人物/数字
- 例：「2026-03-28，文案师在优化产品描述时，发现【删除形容词】能提升转化率 8%」

**❌ "见过"段禁止**：
- 「曾经见过有人…」（无据可查）
- 「通常…」（泛化）
- 「一般来说…」（模糊）

**✅ "因此知道"段必须包含**：
- 明确的因果推导链
- 例：「因为用户首先被【强卖点】吸引，所以减少形容词能让重点更突出」

**❌ "因此知道"段禁止**：
- 「最佳实践是…」（跳过推导）
- 「应该…」（直接命令）

**✅ "现在会"段必须包含**：
- 具体的下一步**动作**
- 例：「下一次在需求阶段就会询问『是否以【强卖点】为重点』」

**❌ "现在会"段禁止**：
- 「保持谨慎」（抽象）
- 「注意…」（模糊）

**质量标准**：
- ✅ EXPERIENCE.md 第一部分完整（3～5条结构性经验）
- ✅ 所有经验符合"见过→因此知道→现在会"三段格式
- ✅ SKILL.md 包含激活读取指令
- ✅ 目录结构规范

**禁止行为**：
- ❌ 禁止省略激活读取指令
- ❌ 禁止在没有充分理由的情况下生成超过 5 条结构性经验
- ❌ 禁止编造不存在的经验

### 技能 4：Skill Auditor（SKILL包→审计报告）

**职责**：对 Stage 3 生成的完整SKILL包进行五维度审计

**输入**：
- SKILL 包（来自 Stage 3）
- 包含：SKILL.md / EXPERIENCE.md / scripts/ / references/

**输出**：
```
审计报告：
- 综合评级：A/B/C/D/E
- 质量评分：N/100
- 安全评分：N/100
- 规范评分：N/100
- 自动修复项：N 项
- 需确认项：N 项
- 修复建议清单
```

**执行流程**：

```
Step 1：结构验证
  检查项：
  ├─ 必需文件存在（SKILL.md）
  ├─ 目录结构符合规范
  ├─ 文件命名合规
  └─ 无多余/临时文件

Step 2：内容分析
  检查项：
  ├─ SKILL.md格式完整（frontmatter / 步骤 / 示例）
  ├─ EXPERIENCE.md结构正确
  ├─ 代码逻辑清晰（如有脚本）
  └─ 跨文件引用一致

Step 3：安全扫描
  检查项：
  ├─ 无硬编码凭证
  ├─ 无危险函数调用
  ├─ 无系统权限越界
  └─ 隐私保护完整

Step 4：规范性验证
  检查项：
  ├─ 命名规范（变量/函数/文件）
  ├─ 文档完整性
  ├─ 依赖管理规范
  └─ YAML格式正确

Step 5：自动修复或确认
  自动修复（无需用户确认）：
  - 缺少frontmatter字段 → 补充默认值
  - 步骤编号格式错误 → 修正
  - 拼写/标点错误 → 修正
  
  需用户确认的修复：
  - 步骤顺序变更
  - 新增/删除功能
  - 核心逻辑修改

Step 6：生成审计报告
  展示：
  ├─ 综合评级
  ├─ 三个评分（质量/安全/规范）
  ├─ 自动修复结果
  └─ 需确认项清单
```

**质量标准**：
- ✅ 评级合理（A ≥ 80 / B ≥ 70 / C ≥ 60 / D ≥ 40 / E < 40）
- ✅ 修复建议具体可执行
- ✅ 无遗漏的严重问题
- ✅ 报告清晰易理解

**禁止行为**：
- ❌ 禁止隐瞒严重问题只显示评分
- ❌ 禁止给出模糊的修复建议
- ❌ 禁止对评级E的SKILL直接通过

## 完整流程

### 端到端流程图

```
用户请求
   │
   ↓
┌─────────────────────────────────────────────────────────────┐
│ Stage 0: 依赖检查                                           │
│ 确保三个技能已安装                                          │
└─────────────────────────────────────────────────────────────┘
   │ ✓通过 / ✗失败
   ├─→ ✗ [停止，提示安装]
   │
   ↓
┌─────────────────────────────────────────────────────────────┐
│ Stage 1: 需求挖掘 (prompt-generator)                       │
│ 输入：用户描述                                              │
│ 输出：SKILL.md草稿                                          │
└─────────────────────────────────────────────────────────────┘
   │ 生成完成
   ↓
用户确认需求摘要
   │ ✓确认 / ✗修改
   ├─→ ✗ [返回Stage 1重新询问]
   │
   ↓
┌─────────────────────────────────────────────────────────────┐
│ Stage 2: 审核提示词 (prompt-auditor)                       │
│ 输入：SKILL.md草稿                                          │
│ 输出：审核意见（问题清单 + 修复建议）                      │
└─────────────────────────────────────────────────────────────┘
   │ 审核结果
   ├─→ P0严重 [回到Stage 1]
   ├─→ P1重要 [就地修复 → 再审一次]
   ├─→ P2优化 [直接修复 → 继续]
   │
   ↓ ✓审核通过
┌─────────────────────────────────────────────────────────────┐
│ Stage 3: 生成SKILL包 (skill-creator)                       │
│ 输入：审核通过的SKILL.md                                   │
│ 输出：完整SKILL包（SKILL.md + EXPERIENCE.md + scripts/ +  │
│       references/）写入临时目录                            │
└─────────────────────────────────────────────────────────────┘
   │ 生成完成
   ↓
┌─────────────────────────────────────────────────────────────┐
│ Stage 4: 审计并修复 (skill-auditor)                        │
│ 输入：完整SKILL包                                           │
│ 输出：审计报告（评级 + 修复建议）                          │
└─────────────────────────────────────────────────────────────┘
   │ 审计结果
   ├─→ 评级A/B/C [询问是否安装]
   ├─→ 评级D [询问是否放弃或继续修复]
   ├─→ 评级E [强烈建议放弃，或继续修复]
   │
   ↓ 用户确认安装
┌─────────────────────────────────────────────────────────────┐
│ Stage 5: 安装 (系统)                                        │
│ 动作：                                                      │
│ 1. 从临时目录移动到正式目录                                │
│ 2. 如指定员工，更新员工配置                                │
│ 3. 询问是否有值得记录的时刻                                │
└─────────────────────────────────────────────────────────────┘
   │ 用户反馈
   ├─→ "有" [追加到EXPERIENCE.md第二部分]
   ├─→ "没有" [保留空占位符]
   │
   ↓
🎉 SKILL生产完成
```

### 关键决策点

| 节点 | 判断条件 | 是 | 否 |
|------|---------|----|----|
| 依赖检查 | 三个技能存在 | 继续 | 提示安装 |
| 需求确认 | 用户同意摘要 | 生成提示词 | 回到询问 |
| 审核通过 | P0=0且P1修复完 | 继续 | 回到需求 |
| 自动修复 | 低级问题 | 直接修复 | 询问用户 |
| 安装决策 | 用户同意 | 移动文件 | 保留临时 |
| 经验记录 | 用户有反馈 | 追加第二部分 | 保留空占位符 |

## EXPERIENCE.md 加载机制

### 设计理念

**EXPERIENCE.md 是 SKILL 的"成长记录"，包含两部分：**

- **第一部分**（结构性经验）：制作时写入，代表该 SKILL 应该遵守的原则和该职能域的最佳实践
- **第二部分**（自然积累）：使用过程中追加，代表这个 SKILL 在真实场景中学到的东西

**每次 SKILL 被激活时，自动读取 EXPERIENCE.md，作为隐式判断依据影响执行。**

### 激活时的加载流程

```
SKILL被调用
   │
   ↓
【激活时必读】段落触发
   │
   ├─ 寻找EXPERIENCE.md
   │   ├─ 找到 → 读取
   │   └─ 未找到 → 跳过，继续执行
   │
   ↓
读取EXPERIENCE.md的所有EXP条目
   │
   ├─ 提取"见过"（背景/约束）
   ├─ 提取"因此知道"（判断规则）
   └─ 提取"现在会"（行动指南）
   │
   ↓
【静默加载】— 不对用户显示读取过程
   │
   └─ 将EXP条目作为隐式判断依据，直接影响本次执行决策
```

### 使用场景示例

#### 示例 1：文案生成 SKILL 的 EXPERIENCE.md

```markdown
## EXP-001：表情符号的陷阱

**见过**：2026-03-18，用户要求"生成小红书文案，风格活泼"，生成了6个😂😍表情，用户说"太夸张了"

**因此知道**：表情符号的"活泼度"是0-1之间的连续变量，不是0就是1。用户说"活泼"不等于要求满屏表情

**现在会**：第一次生成时只加0-2个表情，在用户反馈后再调整，而不是一开始就过度装饰

## EXP-002：用户"没说"不等于"不需要"

**见过**：2026-03-19，用户没提"字数要求"，我生成了800字，用户说"太长了，要300字"。发现这是【开放平台】的默认限制

**因此知道**：在【开放平台】类场景中，如果用户没指定字数，需要主动确认而不是随意生成

**现在会**：生成前主动追问"字数有没有限制？"，特别是在【平台受限】的场景
```

当这个 SKILL 被激活时：

```
第一次激活（无用户字数指定）：
  → 加载EXP-002
  → 主动追问"字数限制是多少？"
  → 用户说"300字"
  → 生成300字文案

第二次激活（用户没说要表情）：
  → 加载EXP-001
  → 只加0-2个表情（不过度装饰）
  → 等待用户反馈再调整
```

#### 示例 2：数据分析 SKILL 的 EXPERIENCE.md

```markdown
## EXP-001：高海拔地区的传感器失效

**见过**：2026-03-22，用户上传的Garmin数据显示【心率突然升到180】，经验证是在西藏高原跑步，实际心率在150左右

**因此知道**：高海拔环境（>3000m）会导致气压传感器和心率算法产生显著偏差，不能直接用平原的计算公式

**现在会**：分析前主动询问"运动地点的海拔是多少？"，如果是高海拔就启用【高海拔校准模式】
```

当这个 SKILL 被激活时：

```
用户上传新数据：
  → 加载EXP-001
  → 主动询问"运动地点的海拔？"
  → 用户说"青岛海边"（低海拔）
  → 使用标准分析算法
  
或者用户说"武汉" → 触发高海拔检测 → 启用校准模式
```

### 经验加载的工作机制

#### 工作模式：隐式影响

**原则**：加载经验后，**不显式告知用户**，而是直接改变 SKILL 的行为

```
用户："帮我生成一份小红书文案"

【不应该这样做】：
  AI: "我读取了EXPERIENCE.md中的2条经验..."
  这样太冗长，破坏用户体验

【应该这样做】：
  AI: "好的，在生成前我想确认两个问题：
       1. 字数有没有限制？
       2. 需要多少表情符号（0-3个范围）？"
  → 这些问题的来源是EXP，但用户不需要知道
```

#### 工作流程

```
SKILL激活
  │
  ├─ 读取EXPERIENCE.md → 提取EXP条目（静默）
  │   ├─ 解析"因此知道"→ 得到【判断规则】
  │   └─ 解析"现在会"→ 得到【行动指南】
  │
  ├─ 应用判断规则
  │   ├─ 例：需不需要主动追问某个问题？
  │   ├─ 例：需不需要启用某种特殊模式？
  │   └─ 例：需不需要在某个环节加强验证？
  │
  ├─ 按【行动指南】执行
  │   ├─ 例：问题的具体措辞和顺序
  │   ├─ 例：特殊场景的处理流程
  │   └─ 例：边界条件的检查方式
  │
  ↓
用户看到的只是更聪明的SKILL，看不到背后的经验驱动
```

### EXPERIENCE.md 的动态更新

#### 更新触发条件

| 触发条件 | 例子 | 动作 |
|---------|------|------|
| 用户发现BUG | "你漏了…" | 追加为EXP |
| 用户特殊偏好 | "我喜欢…风格" | 追加为EXP |
| 特殊场景处理 | "在【平台X】要这样做" | 追加为EXP |
| SKILL自我纠正 | "我之前的方案不对" | 升级现有EXP |
| 用户验证 | "正好！我就是这个需求" | 确认EXP有效 |

#### 追加规则

**✅ 可以追加**：
- 用户纠正过我，产生了新理解
- 发现了一个一直没想到的场景
- 某条经验在真实使用中被验证有效
- 发现了需要特殊处理的边界条件

**❌ 禁止追加**：
- "这次任务顺利完成了" → 顺利不是经验
- "用户对结果满意" → 满意也不是经验
- "重复之前已有的经验"
- "基于假设而不是真实发生的事"

#### 追加格式（与第一部分完全一致）

```markdown
## EXP-新编号：[简短标题]

**见过**：[真实发生的具体情况]
**因此知道**：[推导出的规则]
**现在会**：[具体的行动方案]

> 追加时间：2026-03-25
> 来源：用户反馈 / 自我纠正
```

#### 更新版本号

每追加一条EXP，更新SKILL.md的版本号：

```
version: 1.0.0  →  version: 1.0.1  （追加一条EXP）
version: 1.0.1  →  version: 1.0.2  （再追加一条EXP）
version: 1.0.2  →  version: 1.1.0  （发现重要缺陷，修复后追加）
```

## 总结

| 维度 | 说明 |
|------|------|
| **四技能分工** | 需求→生成→审计→创建 |
| **关键流程** | Stage 1→2→3→4→5，每阶段有明确的通过条件 |

| **用户参与** | 需求→经验来源选择→反馈确认，完全可控 |
| **持续优化** | 经验自然积累，版本号递增，SKILL越用越聪明 |
