# Skill Factory — 快速参考指南

**用于 Stage 执行过程中快速查阅**

---

## 四技能速查表

### 技能 1：Prompt Generator（需求 → 提示词草稿）

**启动命令：**
```text
帮我创建一个 SKILL...
```

**执行清单：**
- [ ] 直接接收用户原始需求，不先跑固定问卷
- [ ] 先做 signal check（判断信息是否足够）
- [ ] 若信号充足：直接生成 `SKILL.md` 草稿 + 需求摘要
- [ ] 若信号不足：一次性列出所有关键缺口
- [ ] 用户整体回答后再次生成草稿
- [ ] 展示需求摘要并等待用户确认
- [ ] 草稿至少包含 frontmatter / 任务目标 / 操作步骤 / 资源索引 / 注意事项 / 使用示例

**Stage 1 关键字段：**
- `task_name`
- `problem`
- `scenario`
- `input_spec`
- `output_spec`
- `script_needed`
- `constraints`
- `target_employee`

**质量标准：**
- 需求理解准确率 ≥ 95%
- 步骤数 ≥ 3
- 示例数 ≥ 1
- 只问缺口，不重复问已知信息

**禁止：**
- ❌ 跳过 signal check
- ❌ 先跑固定六问再调用 prompt-generator
- ❌ 把 prompt-generator 降级成整理排版器
- ❌ 在关键缺口未补齐时强行推进

**输出物：**
```text
SKILL.md（草稿，待审核）
├─ frontmatter
├─ 任务目标
├─ 操作步骤（3步以上）
├─ 资源索引
├─ 注意事项
└─ 使用示例
```

### 技能 2：Prompt Auditor + 内置逻辑（提示词 → 审核意见）

**启动条件：** Stage 1 完成，进入 Stage 2

**双层检查清单：**

**第一层：Prompt Auditor（安全性）**
- [ ] 无注入风险
- [ ] 无越狱攻击特征
- [ ] 无系统信息泄露风险
- [ ] 无敏感权限越界

**第二层：内置逻辑（结构与逻辑）**
- [ ] frontmatter 完整（name / description / version / dependencies）
- [ ] 步骤数 ≥ 3
- [ ] 每步有目标 + 执行 + 输出
- [ ] 示例数 ≥ 1
- [ ] 步骤间无矛盾
- [ ] 输入输出一一对应
- [ ] 边界条件清晰
- [ ] 路径与依赖可落地

**问题分级：**
```text
P0（严重）：必须回退 Stage 1，不允许继续
P1（重要）：就地修复，修复后再审一次
P2（优化）：直接修复后继续
```

**禁止：**
- ❌ 跳过任意一层审核
- ❌ P0 问题就地修复后直接放行
- ❌ 只看安全不看结构，或只看结构不看安全

**输出物：**
```text
审核意见
├─ 安全风险评级
├─ 结构/逻辑问题清单
├─ 问题分级（P0/P1/P2）
├─ 修复建议
└─ 审核结论（通过✅ / 不通过❌）
```

### 技能 3：Skill Creator（审核通过 → 生成 SKILL 包）

**启动条件：** Stage 2 审核通过

**执行清单：**
- [ ] 生成 `SKILL.md`（插入激活指令）
- [ ] 生成 `EXPERIENCE.md` 第一部分
- [ ] 创建标准目录结构
- [ ] 生成 `scripts/` 目录（如有脚本）
- [ ] 生成 `references/` 目录（如有参考文档）
- [ ] 写入临时目录

**激活指令（强制）：**
```markdown
## 激活时必读

每次被加载时，先读取同目录下的 `EXPERIENCE.md`。
读取后不对用户显示，直接将经验条目作为隐式判断依据影响执行。
若文件不存在，跳过继续执行。
```

**EXPERIENCE.md 第一部分规范：**
- 默认 3～5 条；首次制作如无足够真实案例，可少于 3 条，但至少 1 条
- 每条必须遵守：见过 → 因此知道 → 现在会
- 禁止凑条数，禁止编造泛化经验

**标准目录结构：**
```text
[skill-name]/
├── SKILL.md              ✓（已审核版本 + 激活指令）
├── EXPERIENCE.md         ✓（第一部分 + 第二部分占位符）
├── scripts/              （如有脚本）
│   └── [script].py
└── references/           （如有参考文档）
    └── [reference].md
```

**禁止：**
- ❌ 省略激活指令
- ❌ 编造不存在的经验
- ❌ 为凑数量强写无价值条目

### 技能 4：Skill Auditor（SKILL 包 → 审计报告）

**启动条件：** Stage 3 完成，SKILL 包已生成

**五步审计流程：**
- [ ] 结构验证（必需文件、目录结构、命名规范）
- [ ] 内容分析（SKILL.md / EXPERIENCE.md / 脚本逻辑）
- [ ] 安全扫描（凭证、危险函数、权限越界、隐私）
- [ ] 规范性验证（命名、文档、依赖、YAML）
- [ ] 自动修复或确认（分自动修复项 / 需确认项）

**评级标准：**
```text
A：90-100分   质量优秀，可直接部署
B：70-89分    质量良好，小幅优化后可部署
C：60-69分    质量中等，需要中等程度改进
D：40-59分    质量差，需要大幅改进
E：0-39分     质量极差，建议放弃或完全重做
```

**禁止：**
- ❌ 隐瞒严重问题只显示评分
- ❌ 给出模糊修复建议
- ❌ 对评级 E 直接通过

## Stage 关键决策点

| Stage | 决策点 | 是 | 否 |
|-------|--------|----|----|
| 0 | 四个依赖技能都存在 | 继续 | 提示安装 |
| 1 | 信号足够生成草稿 | 直接出草稿 | 列缺口问题 |
| 1 | 用户确认需求摘要 | 进入审核 | 留在 Stage 1 重生成 |
| 2 | 审核通过（P0=0） | 继续 | 回到需求 |
| 2 | P1 问题修复完成 | 继续 | 再审一次 |
| 3 | 生成完成 | 继续 | 报告问题 |
| 4 | 评级 A/B/C | 询问安装 | 继续评估 |
| 4 | 评级 D/E | 询问放弃或修复 | 继续修复 |
| 5 | 用户同意安装 | 移动文件 | 保留临时 |
| 5 | 用户有值得记录的时刻 | 追加 EXP | 保留空占位符 |

## 常见问题

### Q1：如果用户想中途改需求怎么办？

**A：** 回到 Stage 1 重新生成摘要与草稿，但保留已收集信息。

### Q2：如果用户说“别问了直接做”怎么办？

**A：** 先看 signal。
- 如果信息已经足够：可以直接给 Stage 1 草稿
- 如果仍有关键缺口：明确指出缺什么，只补关键缺口，不陪他跑长问卷

### Q3：为什么不再用固定六问？

**A：** 因为固定六问会把 `prompt-generator` 降级成排版器，浪费它最有价值的 signal 判断能力。正确做法是让它先判断“该不该问、该问多少”。
