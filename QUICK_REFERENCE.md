# Skill Factory — 快速参考指南

**用于Stage执行过程中快速查阅**

## 四技能速查表

### 技能1：Prompt Generator（需求→提示词）

**启动命令**：
```
帮我创建一个SKILL...
```

**执行清单**：
- [ ] Q1: 这个SKILL要解决什么问题？
- [ ] Q2: 什么情况下会用到它？
- [ ] Q3: 使用者输入什么？期望得到什么输出？
- [ ] Q4: 是否需要脚本支持？
- [ ] Q5: 有没有特殊约束或限制？
- [ ] Q6: 绑定给哪个员工？
- [ ] 追问确认：还有其他要求吗？
- [ ] 需求摘要确认
- [ ] 生成SKILL.md草稿

**质量标准**：
- 需求理解准确率 ≥ 95%
- 步骤数 ≥ 3
- 示例数 ≥ 1

**禁止**：
- ❌ 跳过需求询问
- ❌ 一次性问完所有问题
- ❌ 在用户未确认前生成

**输出物**：
```
SKILL.md（草稿，待审核）
├─ frontmatter
├─ 任务目标
├─ 操作步骤（3步以上）
├─ 资源索引
├─ 注意事项
└─ 使用示例
```

### 技能2：Prompt Auditor（提示词→审核意见）

**启动条件**：Stage 1完成，进入Stage 2

**四维度检查清单**：

**维度1：结构完整性**
- [ ] frontmatter完整（name/description/version/dependencies）
- [ ] 步骤数 ≥ 3
- [ ] 每步有目标+执行+输出
- [ ] 示例数 ≥ 1

**维度2：安全性**
- [ ] 无注入风险
- [ ] 无越狱攻击特征
- [ ] 无系统信息泄露风险
- [ ] 无敏感权限越界

**维度3：逻辑一致性**
- [ ] 步骤间无矛盾
- [ ] 输入输出一一对应
- [ ] 边界条件清晰
- [ ] 异常处理完整

**维度4：可执行性**
- [ ] 路径正确可访问
- [ ] 依赖声明完整
- [ ] 步骤落地可执行
- [ ] 无模糊指令

### 技能2：内置逻辑（提示词 → 审核意见）

⚠️ **说明**：Stage 2 使用**内置逻辑**对 SKILL.md 草稿进行结构完整性和逻辑一致性审核。

**不使用 Prompt Auditor 的原因**：
- `prompt-auditor` 职责 = 提示词安全注入检测（越狱、系统信息泄露）
- Stage 2 职责 = SKILL.md 结构完整性和逻辑检查（frontmatter、步骤数、示例完整度）
- 两者检查维度完全不同，不存在功能重叠

**启动条件**：Stage 1 完成，进入 Stage 2

**执行清单**：
- [ ] 检查 frontmatter 完整性
- [ ] 检查步骤数 ≥ 3
- [ ] 检查示例数 ≥ 1
- [ ] 检查逻辑一致性
- [ ] 发现问题并分级（P0/P1/P2）
- [ ] 根据问题级别决策（回退/修复/继续）

**问题分级**：
```
P0（严重）：必须回退Stage 1，不允许继续
  例：核心逻辑缺失、安全漏洞、无法执行

P1（重要）：就地修复，修复后再审一次
  例：缺少边界条件、表述不清、依赖不完整

P2（优化）：直接修复后继续
  例：拼写错误、格式不规范、建议优化
```

**禁止**：
- ❌ 跳过某个维度检查
- ❌ P0问题进行就地修复

**输出物**：
```
审核意见
├─ 问题清单（若无则为空）
├─ 问题分级
├─ 修复建议
└─ 审核结论（通过✅/不通过❌）
```

### 技能3：Skill Creator（审核通过→生成SKILL包）

**启动条件**：Stage 2审核通过

**执行清单**：
- [ ] 生成SKILL.md（插入激活指令）
- [ ] 生成EXPERIENCE.md 第一部分
- [ ] 创建标准目录结构
- [ ] 生成scripts/目录（如有脚本）
- [ ] 生成references/目录（如有参考文档）
- [ ] 写入临时目录

**激活指令（强制）**：
```markdown
## 激活时必读

每次被加载时，先读取同目录下的EXPERIENCE.md。
读取后不对用户显示，直接将经验条目作为隐式判断依据影响执行。
若文件不存在，跳过继续执行。
```

**EXPERIENCE.md 第一部分规范**：

写入数量：3～5条

每条格式：
```markdown
## EXP-[编号]：[简短标题]

**见过**：[具体场景——人物/环境/数量，禁止泛化]
**因此知道**：[因果推导，禁止直接跳结论]
**现在会**：[具体可执行行为，禁止抽象原则]
```

**✅ "见过"必须包含**：
- 具体的时间/场景/人物/数字
- 例：「2026-03-28，文案师在优化产品描述时，发现【删除形容词】能提升转化率8%」

**❌ "见过"禁止**：
- 「曾经见过有人…」（无据可查）
- 「通常…」（泛化）
- 「一般来说…」（模糊）

**✅ "因此知道"必须包含**：
- 明确的因果推导链
- 例：「因为用户首先被【强卖点】吸引，所以减少形容词能让重点更突出」

**❌ "因此知道"禁止**：
- 「最佳实践是…」（跳过推导）
- 「应该…」（直接命令）

**✅ "现在会"必须包含**：
- 具体的下一步**动作**
- 例：「下一次在需求阶段就会询问『是否以【强卖点】为重点』」

**❌ "现在会"禁止**：
- 「保持谨慎」（抽象）
- 「注意…」（模糊）

**标准目录结构**：
```
[skill-name]/
├── SKILL.md              ✓（已审核版本+激活指令）
├── EXPERIENCE.md         ✓（第一部分+第二部分占位符）
├── scripts/              （如有脚本）
│   └── [script].py
└── references/           （如有参考文档）
    └── [reference].md
```

**禁止**：
- ❌ 省略激活指令
- ❌ 编造不存在的经验
- ❌ 超过5条结构性经验

**输出物**：
```
完整SKILL包（写入临时目录）
$WORKBUDDY_SKILLS_DIR/temp/[skill-name]/
├── SKILL.md
├── EXPERIENCE.md
├── scripts/
└── references/
```

### 技能4：Skill Auditor（SKILL包→审计报告）

**启动条件**：Stage 3完成，SKILL包已生成

**五步审计流程**：

**Step 1：结构验证**
- [ ] 必需文件存在（SKILL.md）
- [ ] 目录结构符合规范
- [ ] 文件命名合规
- [ ] 无多余/临时文件

**Step 2：内容分析**
- [ ] SKILL.md格式完整
- [ ] EXPERIENCE.md结构正确
- [ ] 代码逻辑清晰（如有脚本）
- [ ] 跨文件引用一致

**Step 3：安全扫描**
- [ ] 无硬编码凭证
- [ ] 无危险函数调用
- [ ] 无系统权限越界
- [ ] 隐私保护完整

**Step 4：规范性验证**
- [ ] 命名规范（变量/函数/文件）
- [ ] 文档完整性
- [ ] 依赖管理规范
- [ ] YAML格式正确

**Step 5：自动修复或确认**

自动修复（无需确认）：
- 缺少frontmatter字段 → 补充默认值
- 步骤编号格式错误 → 修正
- 拼写/标点错误 → 修正

需确认的修复：
- 步骤顺序变更
- 新增/删除功能
- 核心逻辑修改

**评级标准**：
```
A：90-100分   质量优秀，可直接部署
B：70-89分    质量良好，小幅优化后可部署
C：60-69分    质量中等，需要中等程度改进
D：40-59分    质量差，需要大幅改进
E：0-39分     质量极差，建议放弃或完全重做
```

**禁止**：
- ❌ 隐瞒严重问题只显示评分
- ❌ 给出模糊修复建议
- ❌ 对评级E直接通过

**输出物**：
```
审计报告
├─ 综合评级：A/B/C/D/E
├─ 质量评分：N/100
├─ 安全评分：N/100
├─ 规范评分：N/100
├─ 自动修复项：N项
├─ 需确认项：N项
└─ 修复建议清单
```

## Stage 关键决策点

| Stage | 决策点 | 是 | 否 |
|-------|--------|----|----|
| 0 | 三个技能都存在 | 继续 | 提示安装 |
| 1 | 用户确认需求摘要 | 生成提示词 | 回到询问 |
| 2 | 审核通过（P0=0） | 继续 | 回到需求 |
| 2 | P1问题修复完成 | 继续 | 再审一次 |
| 3 | 生成完成 | 继续 | 报告问题 |
| 4 | 评级A/B/C | 询问安装 | 继续评估 |
| 4 | 评级D/E | 询问放弃或修复 | 继续修复 |
| 5 | 用户同意安装 | 移动文件 | 保留临时 |
| 5 | 用户有值得记录的时刻 | 追加EXP | 保留空占位符 |

## EXPERIENCE.md 快速查阅

### 加载时机

```
SKILL被调用
  → 检查是否存在EXPERIENCE.md
  → 存在 → 读取所有EXP条目 → 作为隐式判断依据
  → 不存在 → 跳过继续执行
```

### 加载后的行为改变

```
【不对用户显示】读取过程

只改变SKILL的执行逻辑，用户看到的是更聪明的SKILL

例：
  加载前：直接生成文案
  加载后：先主动问"字数限制是多少？" → 再生成
         （这个问题的来源是EXP，用户不需要知道）
```

### 追加规则（Stage 5后）

**触发条件**：
```
AI问用户："有没有让你觉得『幸亏我们确认了』或『差点踩坑』的时刻？"

用户说"有" → 追加为EXP到第二部分
用户说"没有" → 保留空占位符，不强行追加
```

**追加格式**：
```markdown
## EXP-新编号：[简短标题]

**见过**：[真实发生的具体情况]
**因此知道**：[推导出的规则]
**现在会**：[具体的行动方案]

> 追加时间：YYYY-MM-DD
> 来源：用户反馈 / 自我纠正
```

**禁止追加**：
- ❌ "这次任务顺利完成了"（顺利不是经验）
- ❌ "用户对结果满意"（满意也不是经验）
- ❌ 重复已有的经验
- ❌ 基于假设而非真实发生

## 常见问题

### Q1：如果用户想中途改需求怎么办？

**A**：回到Stage 1重新开始
```
用户："我想修改需求"
AI: "没问题，我们回到Stage 1重新询问。
    之前收集的信息我会保留，只更新变化的部分"
```

### Q2：Stage 2发现P0问题多少次后应该放弃？

**A**：连续3次回退仍有P0问题时
```
如果：
  Stage 2 第1次 → 回退
  Stage 2 第2次 → 再回退
  Stage 2 第3次 → 仍有P0
  
则：暂停，汇报用户，请求人工介入
```

### Q3：EXPERIENCE.md 的"见过"段能不能写"通常"或"一般"？

**A**：不能，必须具体
```
❌ 错误："曾经见过有人在优化文案时遇到问题"
✅ 正确："2026-03-22，用户优化【小红书笔记】时，
          发现7个表情符号导致打开率下降15%"
```

### Q4：如果用户说"我没有特殊要求就跳过吧"？

**A**：明确拒绝，需求确认是强制步骤
```
用户："别问那么多，直接做吧"
AI: "不行，需求确认是我们的强制步骤。
    我理解你急，但这样能确保我们制作的SKILL是你真正需要的。
    我会尽快问完，不会很长。"
```
