# 安全边界声明

> 本流水线所有修改均通过人机对话执行，任何文件写入前需要用户确认。

# Skill Factory — SKILL 自动化生产流水线

---

## 核心指令

当用户说"帮我创建一个 SKILL"或类似请求时，**立即执行本流水线**，禁止跳过任何阶段。

**执行顺序（不可更改）：**

```
Stage 0 → Stage 1 → Stage 2 → Stage 3 → Stage 4 → Stage 5
```

**禁止行为：**
- 禁止跳过需求询问直接生成 SKILL.md
- 禁止在用户未确认需求前进入下一步
- 禁止绕过 Stage 2 和 Stage 4 的审核
- 禁止在未获得用户确认前写入文件

---

## Stage 0：检查依赖（必须首先执行）

**立即执行以下检查：**

1. 解析技能根目录：
   - 优先读取环境变量 `WORKBUDDY_SKILLS_DIR`
   - 未设置时使用：`%USERPROFILE%\.workbuddy\skills\`（Linux/macOS：`~/.workbuddy/skills/`）

2. 检查以下四个目录是否存在：
   - `[根目录]\prompt-generator\`
   - `[根目录]\prompt-auditor\`
   - `[根目录]\skill-creator\`
   - `[根目录]\skill-auditor\`

3. 如有缺失 → 列出缺失列表 → **立即停止，告知用户安装缺失技能后方可继续**

4. 如全部存在 → 宣布"依赖检查通过，开始 Stage 1" → 输出阶段小结

**阶段输出格式：**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ Stage 0 完成 — 依赖检查通过
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
prompt-generator  ✓
prompt-auditor   ✓
skill-creator    ✓
skill-auditor    ✓
→ 开始 Stage 1
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## Stage 1：需求挖掘（强制执行，禁止跳过）

**立即执行以下步骤：**

### 第一步：启动询问

对用户说：

> 「好的，我们开始制作这个 SKILL。我需要先通过几个问题了解你的需求，请逐条回答。」

### 第二步：逐项提问（每问一项，等待回答）

依次提出以下问题，**每问一个必须等待回答，不得连续提问：**

| # | 问题 | 如果用户未回答 |
|---|------|--------------|
| Q1 | 这个 SKILL 要解决什么问题？ | 继续追问具体场景 |
| Q2 | 什么情况下会用到它？ | 继续追问触发条件 |
| Q3 | 使用者输入什么？期望得到什么输出？ | 要求给出具体例子 |
| Q4 | 是否需要 Python/Node 脚本支持？ | 如未说明则默认不需要 |
| Q5 | 有没有特殊约束或限制？ | 如未说明则默认无 |
| Q6 | 绑定给哪个员工？（可选） | 如未说明则默认不绑定 |

### 第三步：追问确认（强制，至少一轮）

第一轮提问结束后，必须追问：

> 「还有其他要求吗？」

如果用户回答模糊 → 继续追问，直到明确为止。

### 第四步：需求理解确认

收集完所有信息后，立即展示需求摘要：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 需求理解确认
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
SKILL 名称：[name]
功能描述：[description]
触发场景：[scenarios]
输入：[input]
输出：[output]
脚本需求：[yes/no + 说明]
目标员工：[employee name 或 无]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
这是你想要的吗？确认后我开始生成提示词草稿。
```

**必须等待用户确认后，才能进入下一步。**

### 第五步：生成提示词草稿

用户确认后，通过对话上下文激活 `prompt-generator` 子技能并执行以下输出：

生成内容必须包含：
- frontmatter（name / description / version / dependencies）
- 任务目标
- 操作步骤（至少 3 步，每步有目标 + 执行内容 + 输出格式）
- 资源索引（如有脚本/references）
- 注意事项
- 使用示例（至少 1 个）

参考格式规范：`references/skill-template.md`

**禁止跳过此步骤。禁止在用户需求未确认时生成任何内容。**

**阶段输出格式：**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ Stage 1 完成 — 需求确认 + 提示词草稿已生成
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
询问轮次：[N] 轮
字数：[N] 字
步骤数：[N] 步
是否含示例：[是/否]
→ 开始 Stage 2 审核
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## Stage 2：审核提示词（强制执行）

**立即执行以下步骤：**

1. 通过对话上下文激活 `prompt-auditor` 子技能

2. 将 Stage 1 输出的 SKILL.md 草稿作为输入

3. 执行四维度审核（参考 `references/audit-checklist.md`）：

   | 维度 | 检查内容 |
   |------|---------|
   | 结构完整性 | frontmatter 完整、步骤≥3步、示例≥1个 |
   | 安全性 | 注入攻击、越狱风险、系统信息泄露 |
   | 逻辑一致性 | 步骤矛盾、输入输出不对应、边界缺失 |
   | 可执行性 | 路径正确、依赖声明、步骤可落地 |

4. 根据审核结果立即执行：

   | 审核结果 | 立即执行 |
   |---------|---------|
   | 发现 P0 问题 | 🔴 **强制回退 Stage 1**，报告具体问题，不允许继续 |
   | 发现 P1 问题 | 🟠 **就地修复草稿**，修复后再次审核，通过才进入 Stage 3 |
   | 发现 P2 问题 | 🟡 **直接修复后继续** |
   | 无问题 | 🟢 **宣布审核通过，进入 Stage 3** |

**禁止绕过审核环节。禁止跳过本阶段直接进入 Stage 3。**

**阶段输出格式：**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ Stage 2 完成 — 提示词审核通过
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
发现问题：[N] 个
P0：[N 个] | P1：[N 个] | P2：[N 个]
处理结果：[全部修复/回退重新生成]
最终状态：[通过 ✅ / 回退 🔄]
→ 开始 Stage 3 生成 SKILL 包
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## Stage 3：生成 SKILL 包（强制执行）

**立即执行以下步骤：**

1. 通过对话上下文激活 `skill-creator` 子技能

2. 基于审核通过的 SKILL.md 内容生成完整 SKILL 包

3. 生成标准目录结构：

   ```
   [skill-name]/
   ├── SKILL.md              # 核心提示词（已审核版本）
   ├── scripts/              # 如有脚本需求
   │   └── [script].py
   └── references/           # 参考文档
       └── [reference].md
   ```

4. **写入临时目录**：`%WORKBUDDY_SKILLS_DIR%\temp\[skill-name]\`
   （Linux/macOS：`$WORKBUDDY_SKILLS_DIR/temp/[skill-name]/`）

5. 报告生成结果

**阶段输出格式：**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ Stage 3 完成 — SKILL 包已生成
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
位置：%WORKBUDDY_SKILLS_DIR%\temp\[skill-name]\
文件清单：
  - SKILL.md ✓
  - scripts/[N 个脚本] ✓ / 无
  - references/[N 个文档] ✓ / 无
→ 开始 Stage 4 审计
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

---

## Stage 4：审计并修复（强制执行）

**立即执行以下步骤：**

1. 通过对话上下文激活 `skill-auditor` 子技能

2. 执行完整五步审计：
   - 结构验证（目录完整性）
   - 代码逻辑分析（如有脚本）
   - 安全扫描（注入/凭证/隐私）
   - 合规性验证（SKILL.md 规范）
   - 生成审计报告

3. 根据自动修复规则处理：

   **立即自动修复（无需确认）：**
   - 缺少 frontmatter 字段 → 补充默认值
   - 步骤编号格式错误 → 修正为"第X步"
   - 拼写错误/标点错误 → 修正
   - 缺少输出格式示例 → 补充模板

   **必须获得用户确认后才能修复：**
   - 逻辑结构变更（步骤顺序、增减步骤）
   - 功能需求调整（新增/删除功能）
   - 安全策略修改（权限范围变化）
   - 任何涉及核心逻辑的修改

4. 输出审计报告并等待用户确认

**禁止跳过本阶段。**

**阶段输出格式：**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📊 Stage 4 审计报告
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
综合评级：[A/B/C/D/E]
质量评分：[N]/100
安全评分：[N]/100
规范评分：[N]/100
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
自动修复：[N] 项（已完成）
需确认项：[N] 项（见下方清单）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[需确认修复项清单...]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## Stage 5：安装（可选，需用户确认）

**立即执行以下步骤：**

1. 询问用户：**「审计完成，是否立即安装到技能目录？」**

2. 用户确认后：
   - 将 SKILL 包从临时目录移动到正式目录：
     - 源：`%WORKBUDDY_SKILLS_DIR%\temp\[skill-name]\`
     - 目标：`%WORKBUDDY_SKILLS_DIR%\[skill-name]\`

3. 如 Stage 0 中指定了目标员工：
   - 更新 `output/ai-office-visualization/agents/configs/[员工名].json`
   - 在 `skills` 数组中追加新技能条目
   - 同步更新 `bridge_server.py` 路由规则中对应员工的 `trigger_keywords`

**最终输出格式：**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🎉 SKILL 生产完成
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
SKILL 名称：[name]
安装路径：%WORKBUDDY_SKILLS_DIR%\[name]\
绑定员工：[员工名 / 未绑定]
耗时：Stage1[Ns] → Stage2[Ns] → Stage3[Ns] → Stage4[Ns]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 异常处理

| 情况 | 立即执行 |
|------|---------|
| Stage 1 用户拒绝回答 | 告知需求不明确无法继续，等待用户补充 |
| Stage 2 发现 P0 问题 | 回退 Stage 1，报告具体漏洞 |
| Stage 2 连续3次回退仍有P0 | 暂停，汇报，请求人工介入 |
| Stage 3 脚本生成失败 | 跳过脚本，标注"脚本待补充"，继续后续步骤 |
| Stage 4 审计评级 D/E | 询问用户是否放弃或继续修复 |
| 用户中途想修改需求 | 回到 Stage 1 重新开始，不丢失已收集信息 |
| 用户说"跳过直接做" | **明确拒绝**：需求确认是强制步骤，不可跳过 |

---

## 注意事项

1. **禁止跳过审核**：Stage 2 和 Stage 4 是质量底线，不可绕过
2. **禁止强制推进**：用户未确认前，禁止进入下一步
3. **禁止直接生成**：未完成需求询问，不得生成任何 SKILL 内容
4. **禁止硬编码路径**：所有路径必须使用 `%WORKBUDDY_SKILLS_DIR%` 环境变量
5. **禁止忽略错误**：P0 问题必须立即回退，不得继续

## 使用示例

### 示例 1：完整流程

**用户输入：**
```
帮我创建一个 SKILL，用来分析微信聊天记录，找出沟通问题和改进建议
```

**流水线执行：**

```
Stage 0: 依赖检查通过
Stage 1: 需求询问开始
  → Q1: 这个 SKILL 要解决什么问题？
  → 用户: 分析微信聊天记录，找出沟通问题
  → Q2: 什么情况下会用到它？
  → 用户: 工作群沟通
  → 还有其他要求吗？
  → 用户: 输出Markdown格式报告
  → 需求确认 → 用户确认
  → 生成 chat-analyzer SKILL.md 草稿
Stage 2: 审核通过
Stage 3: 生成 chat-analyzer/ 目录
Stage 4: 审计通过，评级 B
Stage 5: 安装确认 → 用户确认 → 安装完成
```

### 示例 2：用户想跳过询问

**用户输入：**
```
直接帮我做一个简历分析的 SKILL
```

**正确回应：**

> 「好的，我们来做这个简历分析 SKILL。请先回答几个问题，确保我理解你的需求：
>
> Q1：这个 SKILL 要解决什么问题？
>
> 请告诉我，这样我才能生成准确的内容。」
